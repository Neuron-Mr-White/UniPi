# MCP Adapter — Implementation Plan

## Overview

Build `@pi-unipi/mcp` — an extension that integrates Model Context Protocol (MCP) server management into pi's TUI. Users can browse a cached catalog of 7,800+ MCP servers, add them via split-pane overlay, manage enable/disable per project, and have MCP tools automatically registered as pi ExternalTools. Config uses global/project hierarchy with deep merge.

The spec is thorough. This plan breaks it into 10 discrete tasks ordered by dependency: foundation first (types, constants, config), then bridge (JSON-RPC client, tool translator, registry), then TUI (add overlay, settings overlay), then integration (commands, info-screen, events), and finally documentation.

- completed: Task 2 — Config Manager (Read/Merge/Write)
  - Description: Implement `config/manager.ts` — reads global and project MCP configs, merges them per spec rules, and writes back. Also implement `config/schema.ts` for defaults and validation.
  - Dependencies: Task 1
  - Acceptance Criteria: `loadGlobalConfig()` returns global config or defaults. `loadProjectConfig()` returns project config or null. `resolveServers()` merges correctly: project overrides global at server level, `enabled: false` disables even if defined globally. `saveConfig()` writes with chmod 600. Validation rejects missing command/args. Unit-testable pure functions.
  - Steps:
    1. Create `packages/mcp/src/config/schema.ts` — default McpConfig, McpMetadata values, validation function
    2. Create `packages/mcp/src/config/manager.ts`:
       - `getGlobalConfigDir()` → `~/.unipi/config/mcp/`
       - `getProjectConfigDir(cwd)` → `{cwd}/.unipi/config/mcp/`
       - `loadMcpConfig(dir)` → reads `mcp-config.json` or returns empty default
       - `loadMetadata(dir)` → reads `config.json` or returns empty default
       - `loadAuth(dir)` → reads `auth.json` or returns empty
       - `saveMcpConfig(dir, config)` → writes `mcp-config.json` with chmod 600
       - `saveMetadata(dir, meta)` → writes `config.json`
       - `saveAuth(dir, auth)` → writes `auth.json` with chmod 600
       - `resolveServers(global, project)` → merge per spec rules (project wins on conflict, enabled:false deletes)
       - `mergeEnvWithAuth(serverDef, auth)` → merges auth.json env vars into server def at spawn time
    3. Run typecheck

- completed: Task 4 — MCP JSON-RPC Client
  - Description: Implement `bridge/client.ts` — spawns an MCP server as a child process, performs the JSON-RPC `initialize` handshake, and provides `listTools()` / `callTool()` / `disconnect()` methods. Uses stdio transport (stdin/stdout JSON-RPC messages).
  - Dependencies: Task 1
  - Acceptance Criteria: `McpClient.connect(command, args, env)` spawns process, sends `initialize` request, waits for response. `listTools()` returns tool schemas. `callTool(name, args)` sends `tools/call` and returns result. `disconnect()` sends shutdown, kills process. Handles timeout (10s default), process crash, malformed JSON gracefully. All methods are async with proper error types.
  - Steps:
    1. Create `packages/mcp/src/bridge/client.ts`:
       - `McpClient` class implementing the interface from spec
       - Internal: `sendRequest(method, params)` — writes JSON-RPC to stdin, reads response from stdout by ID correlation
       - Internal: `readMessages()` — reads from stdout line-by-line, parses JSON-RPC, dispatches to pending request handlers or notification handlers
       - `connect(command, args, env)` — uses `child_process.spawn`, sets up stdin/stdout pipes, sends `initialize` with `protocolVersion: "2024-11-05"` and capabilities, waits for response, sends `notifications/initialized`
       - `listTools()` — sends `tools/list`, returns `McpTool[]`
       - `callTool(name, args)` — sends `tools/call` with name and arguments, returns `McpToolResult`
       - `disconnect()` — sends shutdown notification, kills process with SIGTERM, waits briefly, SIGKILL if needed
       - Handle stderr: capture to internal buffer, surface on error
       - Timeout handling: configurable per-request timeout (default 10s from `MCP_DEFAULTS.STARTUP_TIMEOUT_MS`)
    2. Run typecheck

- completed: Task 5 — Tool Translator + Server Registry
  - Description: Implement `bridge/translator.ts` (MCP tool schema → pi ExternalTool) and `bridge/registry.ts` (server lifecycle tracking, enable/disable/restart, status queries). The registry manages all connected servers and their registered tools.
  - Dependencies: Task 4
  - Acceptance Criteria: `translateMcpTool()` produces a valid pi tool with `{serverName}__{toolName}` naming, correct parameters (JSON Schema passthrough), and execute function that delegates to McpClient. `ServerRegistry` tracks server state (starting/running/error/stopped), can `startServer()`, `stopServer()`, `restartServer()`, `getAll()`, `getActive()`, `getFailed()`, `getTotalToolCount()`. Emits MCP lifecycle events via pi.events. Tool registration uses `pi.registerTool()`.
  - Steps:
    1. Create `packages/mcp/src/bridge/translator.ts`:
       - `translateMcpTool(mcpTool, serverName, client)` → pi-compatible tool object
       - Name: `${serverName}__${mcpTool.name}` using `MCP_DEFAULTS.TOOL_NAME_SEPARATOR`
       - Parameters: passthrough MCP `inputSchema` (already JSON Schema, compatible with typebox)
       - Execute: calls `client.callTool(mcpTool.name, params)`, joins content text blocks
       - Error wrapping: includes server name in error message, suggests `/unipi:mcp-settings`
    2. Create `packages/mcp/src/bridge/registry.ts`:
       - `ServerRegistry` class
       - `startServer(resolved)` — creates McpClient, connects, discovers tools, registers with pi, updates state, emits events
       - `stopServer(name)` — disconnects client, unregisters tools, updates state, emits events
       - `restartServer(name)` — stop + start
       - `getAll()` / `getActive()` / `getFailed()` — state queries
       - `getTotalToolCount()` — sum of tools across active servers
       - `getServerState(name)` — returns current ServerState
       - Internal: tracks `Map<string, McpRegistryEntry>` with client, tools, state, pid, error info
    3. Run typecheck

## Risks

1. **MCP protocol variability:** Different MCP servers may implement the protocol slightly differently. Mitigation: test with at least 3 servers (GitHub, Filesystem, Playwright) during Task 10.

2. **Child process management in pi context:** Spawning external processes inside pi's extension runtime may have edge cases (signal handling, stdio buffering). Mitigation: robust timeout handling, stderr capture, SIGTERM→SIGKILL escalation.

3. **Catalog parsing fragility:** The awesome-mcp-servers README format could change. Mitigation: `seed-servers.json` as offline fallback, parser should be lenient (skip unparseable entries rather than fail).

4. **Editor component in split-pane:** The pi-tui `Editor` has been used in single-pane overlays but not in split-pane layouts. Mitigation: follow the existing `ctx.ui.custom()` overlay pattern from info-screen, test Editor embedding early.

5. **Config file permissions:** `chmod 600` may not work on all platforms (Windows). Mitigation: use try/catch, log warning but don't fail.

6. **Open questions from spec:** Server stderr handling, max concurrent servers, auth.json merging. These are design decisions that can be deferred to Task 10 with sensible defaults (stderr → log file, max 20 servers, merge auth.json at spawn time).
